Building a blog that lasts 100 years

June 21, 2026 · 8 min read

Every time I set up a blog, something breaks within two years. A build tool becomes unmaintained. A CDN changes its free tier. A JavaScript framework pivots. A font host starts injecting tracking. The posts survive, but the infrastructure around them rots, and eventually you end up with a site that doesn't render — or worse, requires a three-day yak shave before you can publish again.

I've been through this loop enough times that I decided to think differently about it. What would a blog look like if I built it to last not two years, but a hundred?

The constraint is the design

A browser from 1994 can render plain HTML and CSS. The same browser from 1994 cannot render React, Tailwind, Next.js, or anything that depends on a build step, a CDN, or a JavaScript runtime. This isn't nostalgia — it's an engineering constraint with real consequences for durability.

The constraint collapses into a simple rule: plain HTML and one CSS file. No framework. No build step. No JavaScript. No web fonts fetched from external servers. No npm. No node_modules. No tracking scripts. Nothing that requires a dependency to still be maintained in 2126.

The best long-term bet in software is always the one with the fewest moving parts. A static file served over HTTP has been the same bet since 1991, and it keeps paying.

With modern CSS, system fonts, and a well-structured HTML document, you can build something that looks better than most JavaScript-heavy sites while loading in under 100ms from a cold cache. The CSS for this entire site is under 400 lines. I can read it top to bottom in five minutes and understand exactly what it does.

System fonts are underrated

The default reaction to "system fonts" is that you're stuck with something ugly. This is wrong, and the misunderstanding reveals something about how the industry has been trained to think about design.

System fonts are the fonts the operating system has spent years tuning for readability on that specific screen. The font stack for this site's body text: 

--serif: Charter, "Bitstream Charter", "Sitka Text",
         Cambria, Georgia, "Times New Roman", serif;

Charter on macOS. Cambria on Windows. Georgia everywhere else. All excellent. All tuned for long-form reading. All already on your device, loading in zero milliseconds, with no FOUT, no flash, no flicker.

For UI elements and headings, system-ui is even more direct — it literally resolves to whatever Apple, Microsoft, or Google decided their own users should read system text in. San Francisco on macOS. Segoe UI on Windows. Roboto on Android. They're all excellent choices made by teams who thought about it much harder than I would.

The web font industry has done remarkable work conditioning developers to believe that loading a font from a server is normal, necessary, and benign. It isn't. It's a network dependency, a potential GDPR footnote, a render-blocking request, and a single point of failure for the visual design of every page on your site.

Freeze the URLs now

Every URL I've ever shared and watched die looked fine at the time. Medium posts. Old blog posts from platforms that pivoted or shut down. The content was fine; the infrastructure that gave it an address wasn't.

URL structure is the one thing you can't undo later without breaking every link that's been shared, bookmarked, indexed, or cited. So I set mine in concrete before I wrote a single word:

  • / — home and about
  • /blog/ — the writing index
  • /blog/<slug>/ — one directory per post
  • /principles/ — the living constitution

The /blog/<slug>/ pattern uses a directory with an index.html inside, not a flat slug.html file. This means the URL has no file extension. It resolves correctly on Apache, Nginx, GitHub Pages, Cloudflare Pages, and any future host you can imagine — because all of them understand directory indexing, and none of that logic can change without breaking the entire web.

One stylesheet, one prefix

Every page references /style.css. Not ../style.css. The absolute root path. This matters because relative paths are a function of directory depth, which changes if you ever reorganize. An absolute path is a function of the server root, which doesn't.

The same applies to every asset under /assets/. When this site is served from mohaktripathi.com, /style.css resolves to mohaktripathi.com/style.css regardless of which directory the requesting page lives in.

What "durable" actually means

The web has been around for 35 years. In that time, plain HTML and CSS files have never become unreadable. The <p> tag from 1993 still works today. The CSS I write in 2026 will still render in 2091, because browsers take backwards compatibility more seriously than almost any other piece of software on earth.

JavaScript-heavy sites have a median lifespan of around two to four years before something in the dependency chain becomes unmaintained, a service is discontinued, or the framework's philosophy shifts enough that upgrading is effectively a rewrite.

I don't want to lose words to infrastructure. The whole point of writing is that ideas outlive the moment. If the medium is fragile, the ideas are fragile too.

What publishing looks like in practice

This blog is a directory of HTML files, a stylesheet, and a handful of assets. No package.json. No build command. No CI pipeline to maintain. To publish a new post, I:

  1. Create a new directory under /blog/
  2. Drop an index.html in it
  3. Add a <li> to /blog/index.html
  4. Add an <item> to feed.xml
  5. Commit and push

It deploys in under 30 seconds on GitHub Pages with no build step — because there is nothing to build. The template for any new post is a single HTML file with the same header and footer. You can download it and use it yourself:

post-template.html

The aesthetic argument

I want to make one more point, because the purely technical case makes it sound like this approach involves sacrifice. It doesn't.

A page with system fonts, a 680px reading column, generous whitespace, clear heading hierarchy, and one accent color is a better reading experience than almost anything I've seen built with a component library. The best-read personal sites on the internet — Daring Fireball, Paul Graham's essays, Derek Sivers — are essentially this. They look timeless because they have no framework's fingerprints on them.

The visual design of this site is also the argument the site is making. Clarity over performance isn't just a principle I'm writing about — it's the choice that makes the stylesheet 400 lines instead of 40,000. The medium is the message, and this medium is plain.

One hundred years

A blog that lasts a hundred years is just a blog with no external dependencies and frozen URLs, served from a host that exists. That's it. None of the tech is exotic. The hard part is the restraint: resisting the framework, resisting the feature, resisting the clever build step that solves a problem you don't have.

I wrote this down because I want to be able to find it in 2126. And I think I will.
