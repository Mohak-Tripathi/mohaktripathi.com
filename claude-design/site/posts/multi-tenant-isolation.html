Multi-tenant isolation in Postgres: the version I trust

June 12, 2026 · 11 min read

I'm the only engineer on a B2B product, which means every customer's data lives in the same Postgres database I'm looking at right now. The scariest bug class I can ship isn't downtime or a slow query. It's one tenant seeing another tenant's data. That mistake is quiet, it's irreversible, and it ends companies.

So before I wrote a single feature, I had to decide how tenants are kept apart. This is the model I landed on after trying the alternatives — and, just as importantly, the specific failure I designed it to survive.

Three ways to isolate a tenant

There are really only three shapes, and they trade the same two things against each other: how strong the wall between tenants is, versus how much it costs to operate at the number of tenants you actually have.

ApproachIsolationCost at 1,000 tenants
Database per tenantStrongest1,000 databases to migrate, back up, connect to
Schema per tenantStrong1,000 schemas; migrations fan out; connection churn
Shared schema + tenant_idDepends on your codeOne schema, one migration, one pool

Database-per-tenant has the cleanest story and the worst operational day. A schema change is now a thousand schema changes, each able to fail halfway. As a team of one, that math decides it for me: the shared schema is the only thing I can actually operate. But "depends on your code" is doing a lot of work in that table, and it's exactly the weakness I have to close.

Why I run a shared schema with RLS

Every row carries a tenant_id, and the application always filters by it. That's the obvious 90%. The problem is the other 10%: the day I write a query and forget the WHERE tenant_id = $1. The app will happily return everyone's rows, and nothing stops it — because the wall only ever existed in my discipline.

Postgres can move that wall down into the database itself, where forgetting is no longer possible. Row-level security (RLS) makes the filter a property of the table, not the query: 

ALTER TABLE invoices ENABLE ROW LEVEL SECURITY;
ALTER TABLE invoices FORCE ROW LEVEL SECURITY;

CREATE POLICY tenant_isolation ON invoices
  USING (tenant_id = current_setting('app.tenant_id')::uuid);

Now every statement against invoices — SELECT, UPDATE, DELETE — is silently rewritten to include that predicate. FORCE matters: without it the table owner bypasses the policy, and your app's role is often the owner. With it, even I can't write a query that escapes the tenant boundary by accident.

The application filtering by tenant_id is the seatbelt. RLS is the airbag. You wear the seatbelt every time and you still keep the airbag, because the whole point of the airbag is the day you got it wrong.

Setting the tenant on every transaction

The policy reads app.tenant_id from the session, so something has to set it — correctly, on every request, and never leak it to the next one borrowing the same pooled connection. I set it per transaction with SET LOCAL, which resets automatically at commit or rollback: 

func withTenant(ctx context.Context, db *sql.DB, tenantID string, fn func(*sql.Tx) error) error {
	tx, err := db.BeginTx(ctx, nil)
	if err != nil {
		return err
	}
	defer tx.Rollback()

	// SET LOCAL is scoped to this transaction and cleared on commit,
	// so a pooled connection never carries a tenant into the next request.
	if _, err := tx.ExecContext(ctx, "SET LOCAL app.tenant_id = $1", tenantID); err != nil {
		return err
	}
	if err := fn(tx); err != nil {
		return err
	}
	return tx.Commit()
}

One middleware resolves the tenant from the authenticated request and every query runs inside withTenant. There is no code path to the database that skips it — and if one ever sneaks in, RLS sees an unset app.tenant_id and returns nothing rather than everything.

The failure I'm actually guarding against

Not a malicious attacker — me, at 1 a.m., shipping a hotfix with a hand-written query and a missing clause. Defense in depth isn't paranoia about outsiders; it's humility about myself. The app filter will be right almost every time. RLS is there for the one time it isn't, and that one time is the only time that matters.

What I'd tell past me

  • Pick the isolation you can operate, then make it safe — not the safest one you can't run alone.
  • Turn on FORCE ROW LEVEL SECURITY. The default lets the owner role walk straight through the policy.
  • Set the tenant with SET LOCAL inside the transaction, never on the raw connection. Pools reuse connections; sessions leak.
  • An unset tenant should fail closed — zero rows — not fall back to all rows.

None of this is clever, and that's the point. The boundary holds because it stopped depending on me remembering it. Next time I'll write up how I test this — the failing case I run on every deploy to prove a forgotten WHERE really does come back empty.
